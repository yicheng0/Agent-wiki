# 1.RAG是什么？
- RAG(Retrieval Augmented Generation)也就是检索-增强-生成，让大模型基于真实、可更新的知识来回答问题的技术方案
- 不能改变大模型的推理能力、不能改变大模型的输出风格、不能保证100%的准确性

# 2.RAG的工作流程
- 索引阶段(离线) 查询阶段(在线)
- 索引:加载文档->切割文档(Chunking)->向量化(Embedding)->存入向量数据库
- 查询:用户提问->问题向量化->相似度检索->构造增强prompt->大模型生成回答

# 3.微调和RAG各自区别是什么？
- 微调(Fine-tuning)，让大模型具备特定领域的知识，有全量微调、参数高效微调、指令微调
- ![[Pasted image 20260414203724.png]]
- 大多数场景优先选择RAG； 微调适合需要模型输出特定的风格或格式、需要模型学会某种能力、数据量不大且不经常更新，对推理速度有很高要求

# 4.RAG中的文档切割策略
- 大模型上下文有限，需要将文档切成小块
- 固定大小切割、递归字符切割：最常用的切割策略，RecursiveCharacterTextsplitter![[Pasted image 20260414204851.png]]  基于文档结构的切割、语义切割(切割质量最高、计算成本最大)、Agent驱动的智能切割
- 起步阶段：用递归字符切割(RecursiveCharacterTextSplitter),块大小256-512tokens,重叠10%-20%
- 优化阶段：如果文档结构清晰，直接切换到基于文档结构的切割
- 进阶阶段：如果对检索质量有更高要求，尝试语义切割![[Pasted image 20260414205758.png]]

# 5. Re-rank的定义
向量检索返回的结果不一定是最优的，Re-rank就是要解决这个问题的，检索阶段用的Bi-Encoder(双编码器)，即Embedding模型，把查询和文档编码成向量，通过计算向量之间的余弦相似度来判断相关性。

不足之处：把查询和文档分别处理，两个关键缺陷，信息压缩缺失，缺乏查询意图感知

Crose-Encoder：Re-rank使用的是这个交叉编码器，把查询和文档拼在一起，让模型同时看到两者，进行深度交互分析。速度太慢了

两阶段检索：
第一阶段粗排，第二阶段精排
![[Pasted image 20260414211251.png]]

第一阶段用Bi-Encoder快速筛选候选文档， 第二阶段用Cross-Encoder对文档做精细化重排序

主流Re-rank模型:Cohere Reank、BGE-Reanker(开源方案)、ms-marco-MiniLM微软基于MiniLM训练的轻量级Cross-Encoder、Jina Reranker 

实战思路：先检索50到100个候选，再重排序Top5-Top-10，相加比最高
